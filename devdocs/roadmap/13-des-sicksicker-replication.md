# 13. Continuous-time Sick-Sicker replication (DES tutorial)

Reproduce the discrete-event simulation (DES) tutorial of Lopez-Mendez, Goldhaber-Fiebert, and Alarid-Escudero, "A Tutorial on Discrete Event Simulation Models Using a Cost-Effectiveness Analysis Example in R," Medical Decision Making 2026;46(5):533-548 (PMID 42087677). The article simulates the Sick-Sicker model in continuous time from age 25 to 100 with age-dependent background mortality, a Weibull state-residence-time hazard for progression, recurrence, transition rewards, and four strategies, then computes a cost-effectiveness analysis, epidemiological outcomes, and a probabilistic analysis with acceptability curves, expected loss curves, and the expected value of perfect information. It ships as `examples/mdm_des.py` and a website replication tutorial, the fourth entry in the replication gallery.

This item assumes items 3 through 5 (the engines) and extends the framework in four places. Each extension is additive and plugs into the existing contract: draws in, `Outcomes` out.

## Why the continuous-time microsimulation clock, not `DESModel`

The article's simulation kernel samples a latent arrival time for every permitted transition out of the current state, takes the earliest, moves the individual, and repeats. That is exactly what `MicrosimModel(clock="continuous")` does: `event_times` returns sampled times to each competing destination, the engine takes the earliest and accrues continuously between events. The replication therefore uses the continuous microsimulation clock, which is vectorized across individuals and makes the probabilistic analysis tractable. `DESModel` stays the engine for resource-constrained processes (queues, capacity), which the article treats as an extension beyond its base example; the tutorial page states this framing so readers looking for "DES" land on the right engine.

## Extension 1: `LifeTable` (new `heormodel.models.lifetable`)

Age-dependent background mortality needs a sampler for "years until death given the current age," under a hazard ratio for the disease states. No engine support exists; examples so far hard-code a constant rate or embed an age-indexed array in a cohort model.

`LifeTable(ages, rates)` holds piecewise-constant annual mortality rates by age band, the form life tables take, with the last band extended indefinitely so death is certain. `sample_time_to_death(rng, age, hazard_ratio=1.0)` inverts the cumulative hazard exactly: solve `hr * (H(age + t) - H(age)) = e` with `e` standard exponential. Each draw is conditional on survival to the current age, so a `hazards` function can redraw the death time at every state entry, which is how proportional excess mortality (hazard ratios 3 and 10 here) composes with age dependence. `life_expectancy(age, hazard_ratio=1.0)` integrates the survival function band by band and is the analytic mean of the sampler, used in tests. The article samples discretized annual death probabilities with a uniform within-year correction; exact inversion is simpler and statistically exact for piecewise-constant rates, and the tutorial notes this divergence.

## Extension 2: transition amounts accrued over the sojourn

The model pays $1,000 on the Healthy-to-Sick transition, $2,000 on death, and a one-time utility decrement of 0.01 at disease onset. The companion code does not pay these at the event: it adds each to the annual flow of the sojourn that ends in the transition, then multiplies by the discounted sojourn length (matched behavior 1 in the reconciliation note). This is a specialized cost-function convention, not a general engine feature, so the replication reconstructs it from the event history (extension 3) after the run rather than adding an engine hook. `transition_costs_and_utilities` in `examples/mdm_des/transitions.py` multiplies each event's amount by the discounted sojourn integral over `[start, time]` and sums per individual. Validation: an all-exponential variant is a continuous-time Markov chain in which a sojourn-accrued amount enters state `i`'s effective flow as the amount times its transition rate over the state's total exit rate; a test asserts the engine plus this reconstruction matches the closed form.

## Extension 3: an event history and state occupancy over time

The article's postprocessing module B computes survival probabilities, disease prevalence, and dwell-time distributions from the event history. `Outcomes` averages individuals away, so the engine needs a side channel. `MicrosimModel.evaluate` accepts `trace="events"` and returns a long event log with columns `strategy`, `iteration`, `individual`, `time`, `from_state`, `to_state`, one row per transition, on either clock (the discrete clock logs state changes on the cycle grid).

`heormodel.models.state_occupancy(events, states=..., initial_state=..., n_individuals=..., times=...)` returns the proportion of individuals in each state at each requested time, indexed by `(strategy, iteration, time)` with one column per state. Individuals appear in the log only when they move, so the initial state and the population size are explicit arguments. Survival, prevalence, and the transition amounts are all one-line derivations of this table or the raw log, so they live in the example's `outcomes` and `transitions` modules rather than as maintained package functions: survival is one minus the dead-state column, prevalence is the summed disease columns over survival, and dwell times are differences of consecutive event times within an individual.

## Extension 4: expected loss curves in `heormodel.cea`

The article's figure 4C plots expected loss curves, the expected foregone net monetary benefit of choosing each strategy at each willingness-to-pay threshold; the strategy with the lowest expected loss is the frontier choice, and its expected loss equals the expected value of perfect information (EVPI). `expected_loss(outcomes, wtp, effect=None)` returns a DataFrame indexed by the threshold grid with one column per strategy, computed per iteration as the gap to that iteration's best net monetary benefit. A test asserts the identity `expected_loss(...).min(axis=1) == evpi(...)` on the same outcomes. `heormodel.report` gains `plot_expected_loss` beside `plot_ceac`.

## The replication itself

`examples/mdm_des.py` encodes the article's table 1 exactly: onset rate 0.15, recovery rate 0.5, Weibull progression with proportional-hazards scale 0.08 and shape 1.10 on time since entering Sick, US 2015 all-cause mortality rates (the same array `examples/mdm_cohort_timedep.py` embeds) with hazard ratios 3 and 10, strategy A improving the Sick utility 0.75 to 0.95 at $12,000 per year, strategy B scaling the progression hazard by 0.6 at $13,000 per year, AB combining both, treatment costs paid in both disease states, transition rewards as above, and 3% continuous discounting anchored at age 25. The Weibull draw needs no truncation because the engine redraws competing times at every state entry, when time in state is zero; strategy B multiplies the proportional-hazards scale, converted to the sampling parameterization by `scale ** (-1 / shape)`.

The probabilistic analysis uses the article's distributions: gamma for rates and costs, beta for utilities and the onset decrement, and lognormal for the three hazard ratios. To reproduce the companion figures it holds six parameters that the companion draws but never reads at their base case (matched behavior 2): the Weibull progression scale, both treatment costs, both transition costs, and the treated-utility increment. With the scale fixed, only the Weibull shape varies and no scale-shape correlation applies. The article draws 1,000 parameter sets by Latin hypercube sampling; the replication uses plain Monte Carlo through `ParameterSet.sample`, matching the companion code rather than the article's prose.

The article reports no base-case ICER table; its results are figures. Validation is therefore structural plus graphical:

- Cross-validation against a closed form: an all-exponential variant (Weibull shape 1, constant mortality) is a continuous-time Markov chain whose expected discounted costs and QALYs, including the sojourn-accrued transition amounts, solve `(d I - Q) v = r` for generator `Q`. A test asserts the continuous-clock engine plus the amount reconstruction matches this within Monte Carlo error, exercising every new engine path.
- `LifeTable` sampled means match `life_expectancy` analytically (extension 1 tests).
- The tutorial reproduces the article's figure 3 (survival and prevalence by strategy, with SoC and A identical and B and AB identical) and figure 4 (acceptability curves and frontier, expected loss curves, EVPI over a $0 to $200,000 grid at $1,000 steps), and states how the replication's numbers sit relative to the published figures.

## Matching the companion implementation

Reading the companion code against the article surfaced two substantive choices where the companion departs from its own Table 1, each of which moves the published figures. The replication reproduces both, so it matches the figures rather than the specification. A third difference, the random-number seeding, is left as the framework's per-iteration reproducibility guarantee. The full reconciliation, including the minor method differences, is [`replication-notes/mdm-des-departures.md`](../replication-notes/mdm-des-departures.md); the summary follows.

1. Transition amounts accrue over the preceding sojourn. The companion CEA adds `dc_HS1`, `dc_D`, and `du_HS1` to the annual cost and utility rates of the whole sojourn that ends in the transition, then multiplies by the discounted sojourn length, so a $2,000 cost of dying becomes $2,000 per year over the final sojourn. Table 1 defines them as one-time amounts, but the figures follow the code. The replication reconstructs this from the event history in `transition_costs_and_utilities`. Effect: the base case runs about $128,000 to $285,000 in cost, matching figure 4A, roughly $20,000 per strategy above a once-at-event accounting.
2. Six PSA parameters never reach the model in the companion probabilistic analysis. The parameter update is `modifyList`, which matches names exactly, and six drawn columns have names the simulation and cost lists do not read: `r_S1S2_scale` (the simulation reads `r_S1S2_scale_ph`), `c_trtA`, `c_trtB`, `ic_HS1`, and `ic_D` (the cost list reads `dc_trtA`, `dc_trtB`, `dc_HS1`, `dc_D`), and `u_trtA` (the cost list reads a fixed `du_trtA = 0.20` increment on the varying `u_S1`). They stay at base-case values. The replication holds all six fixed. Effect: the acceptability curves are near step functions and the EVPI peaks are small ($2,600 and $750 per person), matching figure 4D.

The one difference left in place: the companion sets the same seed inside every PSA iteration, so all 1,000 parameter sets reuse one random-number stream. The replication gives each iteration its own reproducible stream, the framework's standard design, and relies on common random numbers across strategies within an iteration. This changes the Monte Carlo error structure, not the expected figures, so the replication matches figure 4 within Monte Carlo error.

- `heormodel.models.LifeTable`, `trace="events"` and `heormodel.models.state_occupancy` on the event history, `heormodel.cea.expected_loss`, `heormodel.report.plot_expected_loss`, each with docstring examples and tests.
- `examples/mdm_des/`: a folder of building blocks composed by `run.py`, base case at 100,000 individuals with sojourn-accrued transition amounts, epidemiological outcomes, probabilistic analysis at 1,000 draws holding the six companion-fixed parameters, plots, and a run report.

## Acceptance

- The continuous-time Markov chain cross-validation (engine plus sojourn-accrued amount reconstruction), the life-table analytics, and the expected-loss identity all pass as tests at fixed seeds.
- The example runs under `uv run python examples/mdm_des/run.py`; the rendered tutorial executes; prose matches the printed outputs and plots.
- Survival and prevalence curves show SoC identical to A and B identical to AB, matching the article's figure 3; the acceptability frontier switches SoC to B to AB as the threshold rises, matching figure 4; the cost axis and the small EVPI peaks match figure 4 within Monte Carlo error.
